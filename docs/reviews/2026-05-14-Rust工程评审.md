## 主要问题

### 单会话线程 panic 会拖垮宿主端监听循环

- **分级**：高。
- **问题定性**：违反硬性工程基线。
- **依据**：`review-rust` 要求 `panic!` 限制在测试、初始化不可恢复路径或有明确说明的断言；长驻服务型入口不应让单会话异常升级为进程级崩溃。
- **证据**：`crates/wincast-host/src/agent/listener.rs:204` 与 `listener.rs:245` 调用 `log_session_result(session.join())`；`listener.rs:256` 定义 `log_session_result`；`listener.rs:269` 在 `session.join()` 返回 panic 时直接 `panic!("客户端会话线程异常结束")`。
- **风险**：任一客户端会话线程内部 panic 都会结束宿主端监听循环。后续进入 Windows Service/Agent 常驻形态后，这会把单连接问题扩大为宿主端不可用。
- **建议**：把会话线程 panic 收敛为会话级错误、状态上报和清理结果；监听循环继续运行。必要时将异常上报给 Service 状态机或日志系统，并补充会话线程 panic 后仍可接受下一连接的回归测试。

### D3D readback 的 `Map` / `Unmap` 缺少 RAII 与安全契约

- **分级**：高。
- **问题定性**：违反硬性工程基线。
- **依据**：`review-rust` 要求 unsafe 集中、最小化并写清安全前提；D3D mapped memory 属于真实 FFI 资源生命周期边界。
- **证据**：`crates/wincast-capture/src/windows_impl.rs:279` 调用 D3D `Map`；`windows_impl.rs:283` 用 `Height * RowPitch` 计算长度；`windows_impl.rs:284` 通过 `slice::from_raw_parts(mapped.pData.cast::<u8>(), byte_len)` 读取裸指针；`windows_impl.rs:287` 手动 `Unmap`。当前没有 RAII guard，也没有就地说明 `mapped.pData`、长度、生命周期和 `Unmap` 配对前提。
- **风险**：当前正常路径会释放映射资源，但安全性依赖读者跨 D3D API 约定和控制流推断。后续在 `Map` 与 `Unmap` 之间加入可能 panic 的逻辑或调整长度计算时，容易引入资源泄漏、悬垂读取或越界读取风险。
- **建议**：封装小型 mapped texture guard，在 `Drop` 中执行 `Unmap`；把裸指针转 slice 的范围压在 guard 生命周期内，并用 `// SAFETY:` 注释写清指针有效性、字节长度、row pitch 和 mapped 资源生命周期。

### Windows Service callback context 缺少指针生命周期安全契约

- **分级**：高。
- **问题定性**：违反硬性工程基线。
- **依据**：`review-rust` 要求 unsafe/FFI 边界写清安全前提，尤其是跨系统回调中的裸指针来源、类型和生命周期。
- **证据**：`crates/wincast-host/src/service.rs:460` 创建 `stop_sender`；`service.rs:463` 调用 `RegisterServiceCtrlHandlerExW`；`service.rs:466` 将 `&stop_sender as *const Mutex<mpsc::Sender<()>> as *mut _` 作为 context 传给 SCM；`service.rs:484` 定义 `service_control_handler`；`service.rs:493` 在回调中把 `context` cast 回 `&Mutex<mpsc::Sender<()>>`。
- **风险**：裸指针跨系统回调，Rust 类型系统无法保证 SCM 不会在 `run_service_main` 栈帧结束后继续调用，也无法表达回调线程访问该 mutex 的同步和停止前提。后续调整 Service 主循环、提前返回路径或注册方式时，可能引入悬垂指针或类型不匹配。
- **建议**：补充明确的 `// SAFETY:` 契约，说明 context 的来源、存活时间、回调停止时机和同步前提；或改为堆分配的状态 owner，使释放点和 SCM handler 生命周期可审计。

### `wincast-host` 应用层仍绑在 bin target 的 `main.rs` 上

- **分级**：中。
- **问题定性**：规范缺失导致风险。
- **依据**：`review-rust` 要求 `src/main.rs` 和 bin target 只负责组合与启动，应用层能力应具备可复用边界。
- **证据**：`crates/wincast-host/src/main.rs:10` 起直接声明 `agent`、`agent_runtime`、`program`、`service`、`service_agent`、`service_ipc`、`session_events`、`session_state`、`window` 等模块；同文件还承担默认配置路径、Service 命令分派、配置校验和运行调度，例如 `main.rs:75`、`main.rs:124`、`main.rs:152`、`main.rs:168`。
- **风险**：Host Agent、Service 管理、Service/Agent IPC、会话状态和配置入口都挂在 bin crate 内，后续拆分 Windows Service 与交互桌面 Host Agent、做更高层集成测试或复用运行时边界时，会继续依赖 `main.rs` 的内部模块结构。
- **建议**：将 host 应用层迁入 `src/lib.rs`，按现有职责拆成 `cli/config/service/agent/runtime` 等模块；`main.rs` 保留 clap 入口、错误打印和 `ExitCode` 转换。

### 输入读取和 raw BGRA 读取线程缺少显式 owner、取消和 join 边界

- **分级**：中。
- **问题定性**：规范缺失导致风险。
- **依据**：`review-rust` 要求后台任务、线程、连接和资源生命周期可追踪。
- **证据**：`crates/wincast-host/src/agent/stream.rs:190` 的 `spawn_input_event_reader` 在 `stream.rs:195` `thread::spawn` 后只返回 `mpsc::Receiver<InputReaderEvent>`；`crates/wincast-client/src/render_loop.rs:184` 的 `spawn_raw_bgra_frame_reader` 在 `render_loop.rs:188` `thread::spawn` 后也只返回 receiver。
- **风险**：线程退出依赖 socket 被动关闭或 channel 断开。会话失败、窗口退出、Service 自动恢复、半开连接或重连场景下，线程生命周期不可审计，后续问题会变成难定位的资源泄漏或收尾顺序问题。
- **建议**：为会话级后台线程建立 owner，持有 `JoinHandle`、关闭信号或可中断 stream；会话结束路径统一执行取消、关闭、join 和超时日志。

### Windows 绑定路线同时使用 `windows` 与 `windows-sys`

- **分级**：中。
- **问题定性**：规范缺失导致风险。
- **依据**：`review-rust` 要求检查重复或功能重叠依赖，尤其是平台 API 与 FFI 依赖。
- **证据**：`crates/wincast-capture/Cargo.toml:12` 使用 `windows = "0.62.2"`；`crates/wincast-host/Cargo.toml:15` 和 `crates/wincast-input/Cargo.toml:12` 使用 `windows-sys = "0.61.2"`。
- **风险**：这可能源于 WinRT/WGC 与 Win32 API 的现实差异，但当前 Cargo 与文档没有说明选择边界。后续扩展 FFI、错误类型、句柄类型和升级依赖时，容易出现两套绑定模型混用和审计口径不一致。
- **建议**：明确文档化“捕获层使用 `windows`、SCM/输入使用 `windows-sys`”的架构决策、升级策略和类型转换边界；若没有明确收益，逐步收敛到单一绑定路线。

### 根 `.gitignore` 对本地产物和敏感文件覆盖偏窄

- **分级**：中。
- **问题定性**：规范缺失导致风险。
- **依据**：`review-engineering` 与 `review-rust` 要求 `.gitignore` 覆盖本地环境、密钥、缓存、构建产物、日志、覆盖率和临时文件。
- **证据**：`.gitignore:1` 起当前仅覆盖 `/target/`、`**/*.rs.bk`、根目录 `wincast-host.toml`、`wincast-client.toml`。
- **风险**：Rust 桌面/系统工具开发常见的 `.env`、`*.log`、coverage/profraw、IDE 缓存、dump、临时烟测输出没有兜底规则，后续真机调试和 Service 排查时误提交本地文件或敏感配置的概率较高。
- **建议**：补充项目常见本地产物与敏感文件忽略项；如果团队刻意保持极简 `.gitignore`，需要在贡献说明中明确禁止提交的文件类型。

### 缺少 `.gitattributes`，跨平台换行和二进制规则无仓库级约束

- **分级**：低。
- **问题定性**：规范缺失导致风险。
- **依据**：`review-engineering` 要求检查 `.gitattributes` 是否定义文本换行、二进制文件和 LFS 规则。
- **证据**：仓库根目录未发现 `.gitattributes`；项目同时覆盖 Windows host 与 Linux client，`README.md:10`、`docs/development.md:17` 明确客户端覆盖 Linux x86_64 与 Linux aarch64/ARM64。
- **风险**：不同开发机 `core.autocrlf` 设置可能制造无意义 diff；后续加入截图、二进制样本或安装包时也缺少统一属性规则。
- **建议**：按团队需要补最小文本换行规则和二进制资源规则；若未来引入大体积样本或安装包，再补 LFS 规则。

### Linux SDL2 依赖未纳入 workspace dependencies

- **分级**：低。
- **问题定性**：建议优化。
- **依据**：`review-rust` 建议 workspace dependencies 集中管理共享或重要依赖版本。
- **证据**：根 `Cargo.toml:17` 的 `[workspace.dependencies]` 集中管理 `clap`、`serde`、`serde_json`、`thiserror`、`toml` 和内部 path crates；`crates/wincast-render/Cargo.toml:12` 单独声明 `sdl2 = "0.37"`。
- **风险**：当前只有一个 crate 使用 SDL2，影响有限；但 SDL2 是客户端核心平台依赖，后续测试工具或客户端侧 crate 复用时容易版本分叉。
- **建议**：将 `sdl2` 版本提升到根 `[workspace.dependencies]`，`wincast-render` 使用 `sdl2.workspace = true`。

### 文档缺少只读格式检查入口

- **分级**：低。
- **问题定性**：建议优化。
- **依据**：工程评审和 CI 场景应优先使用 check/dry-run 类命令，不应把只读检查与自动修复混成同一入口。

- **风险**：评审、提交前检查或 CI 文档容易把格式修复和格式检查混用，导致工作区被命令隐式改写。
- **建议**：在开发文档中区分“格式化修复：`cargo fmt --all`”与“只读检查：`cargo fmt --all -- --check`”。

### Rust 工具链未钉扎

- **分级**：低。
- **问题定性**：规范缺失导致风险。
- **依据**：`review-rust` 要求核对 Rust toolchain、edition、MSRV、target 和文档是否一致。

## 整改优先级建议

### 第一阶段：长驻运行稳定性

- 将 `listener.rs` 的会话线程 panic 收敛为会话级错误，不让 host 监听循环因单连接异常退出。
- 为 host 输入读取线程和 client raw BGRA 读取线程补 owner、取消、关闭和 join 边界。
- 为 Windows Service callback context 补安全契约，或改成可审计的堆分配状态 owner。

### 第二阶段：FFI 与 crate 边界

- 为 D3D readback `Map` / `Unmap` 引入 RAII guard，并补 `SAFETY` 注释。
- 将 `wincast-host` 应用层从 bin target 迁到 `src/lib.rs`，让 `main.rs` 只保留入口和退出码转换。
- 明确 `windows` 与 `windows-sys` 双线依赖策略，形成文档化边界或开始收敛。

### 第三阶段：协作与仓库治理

- 扩展 `.gitignore`，补充 `.gitattributes`。
- 将 SDL2 提升到 workspace dependencies。
- 补 `rust-toolchain.toml` 或等效工具链校验说明。
- 在开发文档中区分 `cargo fmt --all` 与 `cargo fmt --all -- --check`。

### 第四阶段：目标平台验证

- 在 Linux x86_64 真机验证 SDL2 链接、窗口创建、渲染和输入回传。
- 在 aarch64/ARM64 Linux 真机执行同一组验证。
- 在 Windows 管理员终端补 Service install/start/stop/uninstall 烟测，并记录限制条件。
