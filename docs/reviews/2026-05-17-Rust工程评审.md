# WinCast Suite Rust 工程评审报告

## 评审范围

本次只评审 Rust 工程体系、Cargo workspace 与 crate 边界、features 与条件编译、错误处理、线程与资源生命周期、unsafe/FFI、依赖治理、仓库治理、质量门禁和文档一致性；不评价功能需求完成度、业务规则正确性或单个功能缺陷。

本次评审以 2026-05-17 当前 `master` 工作树为准。评审过程中未修改业务代码，只新增本评审报告。

## 评审依据

- `review-engineering` 通用工程基线：结论必须有文件、配置、命令输出或仓库状态证据；自动化检查不能把未执行写成通过；入口层应轻量，边界应可复用、可验证；文档、配置和代码应保持一致。

- 项目现场规则：Rust 修改后需要执行 `cargo fmt --all` 和 `cargo clippy --all-targets --all-features -- -D warnings`；除 `cargo fmt --all` 外，`cargo check`、`cargo clippy`、`cargo test`、`cargo build`、`cargo run`、`cargo doc` 等写入 `target` 或运行 build script 的命令需非沙箱执行。

## 主要问题

### 客户端运行逻辑仍绑定在 bin target

- **分级**：中。
- **问题定性**：规范缺失导致风险。
- **依据**：`review-engineering` 要求入口层轻量，业务域、接口适配、配置、类型和测试边界应可区分；`review-rust` 要求 crate 边界和 bin target 职责清楚。
- **证据**：`crates/wincast-client/src/main.rs:3` 至 `main.rs:6` 在 bin 入口内直接声明 `errors`、`render_loop`、`runtime`、`stream` 模块，`main.rs:14` 至 `main.rs:29` 再调用这些私有模块启动客户端；`crates/wincast-client/Cargo.toml:1` 至 `Cargo.toml:11` 只有 package 与 dependencies，没有 lib target。对比 host 侧，`crates/wincast-host/src/main.rs` 已只负责转调 `wincast_host::main()`，运行逻辑位于 lib target。
- **风险**：客户端握手、重试、H.264 读取、渲染循环和错误格式化只能作为二进制内部模块被单元测试覆盖，不能被集成测试、上层 launcher、后续客户端壳层或服务编排以稳定 crate API 复用。后续如果要接打包、托盘或桌面壳层，会继续从 bin 内部模块拆接口。
- **建议**：为 `wincast-client` 增加 `src/lib.rs`，把 `runtime`、`stream`、`render_loop` 和错误格式化收敛为 crate 内清晰 API；`src/main.rs` 只保留参数/默认路径、调用和退出码处理。

### 客户端心跳与视频帧读取耦合

- **分级**：中。
- **问题定性**：规范缺失导致风险。
- **依据**：`review-rust` 要求 channel、task/thread、取消、超时、backpressure 和资源生命周期有清楚策略；长连接输入与心跳不应隐含依赖另一条链路持续产出。
- **证据**：`crates/wincast-client/src/stream.rs:99` 至 `stream.rs:123` 每轮先 `poll_input()`、发送输入和 `Heartbeat`，随后直接阻塞读取下一条 H.264 stream item；同一循环没有独立输入泵、独立 heartbeat owner 或读取超时。host 侧输入 reader 在 `crates/wincast-host/src/agent/stream.rs:390` 至 `stream.rs:399` 为输入流设置读超时并进入独立线程，在 `stream.rs:459` 至 `stream.rs:491` 依赖连续超时计数判断断开。
- **风险**：如果捕获源静止、编码链路卡顿、服务端暂时无帧或网络读阻塞，客户端会停在视频读取上，无法继续泵 SDL 事件和发送心跳；host 输入 reader 可能在连续超时后把会话判定为断开。该风险会影响后续长时间真机烟测、静止画面和异常恢复判断。
- **建议**：把客户端输入/心跳和视频帧读取解耦，例如独立输入线程或带超时的事件循环；同时在协议层明确视频无帧、心跳和输入通道的生命周期关系，并补静止画面或服务端暂时无帧时客户端仍发送心跳的回归测试。

### Host session 事件 API 使用 String 错误

- **分级**：中。
- **问题定性**：规范缺失导致风险。
- **依据**：`review-rust` 要求库 crate 暴露稳定错误类型，避免把文本或内部错误直接扩散到公共 API。
- **证据**：`crates/wincast-host/src/lib.rs:12` 至 `lib.rs:13` 公开 `session_events` 与 `session_state`；`crates/wincast-host/src/session_events.rs:21` 至 `session_events.rs:23` 的 public trait 返回 `Result<DetectedDesktopSession, String>`；`session_events.rs:40` 至 `session_events.rs:42`、`session_events.rs:109` 至 `session_events.rs:127` 的 public 函数也返回 `String` 错误。
- **风险**：后续 Windows Service、Agent 编排或测试工具无法按错误类别处理会话检测失败、平台不支持、WTS 注册失败和注销失败，只能匹配文本；这会让状态机、日志和 UI/CLI 错误展示耦合到中文文案。
- **建议**：为 session 事件层定义 `SessionEventError` 或更窄的错误枚举，用 `thiserror` 保留中文展示文本，同时让调用方能按错误类别分支处理。

### README host 配置示例缺少必填段

- **分级**：中。
- **问题定性**：违反项目已确认规范。
- **依据**：`review-engineering` 要求文档、配置和代码一致；README 是部署入口，示例应能映射到当前配置模型。
- **证据**：`README.md:17` 至 `README.md:22` 的 host TOML 片段只包含 `listen`、`program`、`args` 和 `work_dir`。当前 `HostConfig` 在 `crates/wincast-protocol/src/config.rs:20` 至 `config.rs:28` 要求 `video` 与 `capture` 都是必填字段，完整示例在 `examples/host.toml:6` 至 `examples/host.toml:17` 才包含 `[video]` 和 `[capture]`。
- **风险**：用户按 README 片段复制配置会得到无效 host 配置，入口文档与稳定版配置模型不一致。虽然 README 提到了可从 `examples/host.toml` 复制，但内联示例本身仍会误导部署。
- **建议**：让 README 内联 host 示例直接包含 `[video]` 与 `[capture]`，或删除不完整片段，只保留指向 `examples/host.toml` 的完整示例入口。

### 测试后端作为普通 public API 暴露

- **分级**：低。
- **问题定性**：建议优化。
- **依据**：`review-rust` 要求公共 API、模块和测试工具边界清楚，避免测试工具进入生产路径。
- **证据**：`crates/wincast-media/src/lib.rs:11` 至 `lib.rs:19` 直接 `pub mod test_support`，导出 `FakeH264Decoder`、`FakeH264Encoder` 和 `FAKE_H264_DECODED_PAYLOAD_LIMIT`，没有 `#[cfg(test)]`、feature gate 或 dev-only 边界。
- **风险**：fake H.264 后端进入生产 crate 的公共 API，调用方可能误用测试 payload 或把 fake 后端当作可部署媒体后端；同时扩大了 `wincast-media` 的 semver/API 表面。
- **建议**：如果仅用于 workspace 内测试，将其改为 `#[cfg(any(test, feature = "test-support"))]` 并默认关闭；如果确实要跨 crate 测试复用，应在 feature 和文档中明确它不是生产后端。

## 整改优先级建议

### 第一阶段：修正文档与公共边界

- 修复 README host 配置示例，使入口文档与 `HostConfig` 必填字段一致。
- 为 `wincast-media::test_support` 增加测试专用 feature 或移入更窄的测试辅助边界。

### 第二阶段：收敛客户端 crate 边界

- 增加 `wincast-client` lib target，将 runtime、stream、render loop 和错误格式化从 bin 内部模块迁出。
- 保持 `src/main.rs` 只负责默认路径、退出码和用户可见输出。

### 第三阶段：加固长连接生命周期

- 将客户端输入/心跳循环与视频帧读取解耦，补静止画面、暂时无帧和视频读取阻塞时的心跳回归测试。
- 为 host session 事件层定义稳定错误类型，避免后续 Service/Agent 编排依赖 `String` 文案分支。

### 第四阶段：补目标平台验证

- 在 Linux x86_64 与 aarch64/ARM64 真机补跑 SDL2 + OpenH264 构建、窗口渲染和输入回传烟测。
- 如后续新增打包入口或 Service/Agent 入口，将其纳入同一组 Cargo 与真机验证清单。
