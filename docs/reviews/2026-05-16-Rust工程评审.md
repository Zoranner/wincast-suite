# WinCast Suite Rust 工程评审报告

## 评审范围

本次只评审 Rust 工程体系、Cargo workspace 与 crate 边界、features 与条件编译、错误处理、线程与资源生命周期、unsafe/FFI、依赖治理、仓库治理、质量门禁和文档一致性；不评价功能需求完成度、业务规则正确性或单个功能缺陷。

当前工作树在评审开始前已有多处未提交修改，本报告以当前工作树事实为依据，不回滚、不修复业务代码。

## 评审依据

- `review-engineering` 通用工程基线：结论必须有文件、配置、命令输出或仓库状态证据；自动化检查不能把未执行写成通过；服务型入口不应把局部异常扩大为进程级不可用；仓库治理应覆盖本地配置、日志、构建和测试产物。
- `review-rust` Rust 专项清单：检查 workspace、crate 边界、features、错误处理、async/线程、unsafe/FFI、依赖治理、Cargo 锁文件、fmt/clippy/test/build/doc 和文档同步。
- 项目现场规则：Rust 修改后需要执行 `cargo fmt --all` 和 `cargo clippy --all-targets --all-features -- -D warnings`；除 `cargo fmt --all` 外，`cargo check`、`cargo clippy`、`cargo test`、`cargo build`、`cargo run`、`cargo doc` 等写入 `target` 或运行 build script 的命令需非沙箱执行。
- 当前仓库事实：根 `Cargo.toml`、各 crate `Cargo.toml`、`Cargo.lock`、`.gitignore`、`.gitattributes`、`README.md`、`docs/design.md`、`docs/development.md`、`docs/smoke-test.md`、源码和测试目录。

## 总体结论

WinCast Suite 仍保持清晰的 Rust workspace 主体结构。根 `Cargo.toml:1` 定义 7 个成员 crate，`Cargo.toml:11` 使用 `resolver = "3"`，`Cargo.toml:13` 至 `Cargo.toml:16` 统一声明 Rust 2024 edition、MIT 许可和 `rust-version = "1.88"`；`Cargo.lock` 已提交，符合当前应用/CLI/workspace 的锁文件管理方式。

当前质量门禁没有闭合：`cargo fmt --all -- --check`、`cargo test --workspace --all-targets --all-features`、`cargo build --workspace --all-targets --all-features`、`cargo doc --workspace --all-features --no-deps` 通过，但 `cargo clippy --workspace --all-targets --all-features -- -D warnings` 失败。失败点来自 `crates/wincast-client/src/stream.rs:813` 的手写 `Default`，在 `-D warnings` 下触发 `clippy::derivable_impls`。这属于硬性工程基线失败，不能把当前状态写成静态检查通过。

当前主要风险集中在三处：稳定设计已要求删除 raw BGRA 网络传输，但协议和 host/client 分支仍保留 raw BGRA 公共契约；长驻 host 会话线程 panic 仍会升级为监听侧 panic；OpenH264 测试虽通过，但运行时反复输出码率控制 warning，说明当前编码器参数组合存在后端可观测告警。

## 主要问题

### Clippy 门禁当前失败

- **分级**：高。
- **问题定性**：违反硬性工程基线。
- **依据**：`review-engineering` 要求有明确自动化编译或构建入口时原则上不能有 warning；`review-rust` 要求 clippy warning 作为风险记录，且本项目规则要求 `cargo clippy --all-targets --all-features -- -D warnings` 通过。
- **证据**：`cargo clippy --workspace --all-targets --all-features -- -D warnings` 失败，报错为 `this impl can be derived`；对应 `crates/wincast-client/src/stream.rs:806` 定义 `RecordingRenderer`，`stream.rs:813` 至 `stream.rs:822` 手写 `impl Default for RecordingRenderer`。
- **风险**：当前工作树无法通过项目静态检查门禁。后续提交如果带着该状态进入主线，会让评审、提交和发布前验证无法区分真正新问题与已知 warning。
- **建议**：将 `RecordingRenderer` 改为 `#[derive(Default)]`，或给出不能派生的真实理由。不要用 `#[allow(clippy::derivable_impls)]` 掩盖该问题。

### 设计已删除 raw BGRA 网络链路，但公共协议和运行分支仍保留

- **分级**：高。
- **问题定性**：违反项目已确认规范。
- **依据**：`review-rust` 要求 serde schema、消息契约和代码同步；项目设计文档是当前协议和配置边界依据。
- **证据**：`docs/design.md:34` 写明视频传输使用低延迟 H.264 编码流，不再设计 raw BGRA 网络传输链路；`docs/design.md:145` 写明正式传输链路只保留编码视频流，并删除 raw BGRA 作为网络协议、配置 codec 和烟测主线的概念；`docs/design.md:433` 写明删除 raw BGRA 传输链路，固定 H.264 为第一正式 codec。当前代码仍在 `crates/wincast-protocol/src/lib.rs:6` 公开 `raw_frame` 模块，`crates/wincast-protocol/src/config.rs:96` 至 `config.rs:101` 保留 `VideoCodec::RawBgra`，`crates/wincast-protocol/src/message.rs:15` 保留 `RawBgraReadbackFrame` 控制消息，`crates/wincast-protocol/src/raw_frame.rs:158` 至 `raw_frame.rs:190` 公开 raw BGRA 读写函数；host 在 `crates/wincast-host/src/agent/session.rs:252` 至 `session.rs:261` 仍按 `VideoCodec::RawBgra` 分支进入 raw 流。
- **风险**：协议 crate 的公共契约与稳定设计口径冲突，旧传输路径会继续影响配置校验、测试、客户端兼容判断和后续 API 收敛。`skip_deserializing` 只能阻止 TOML 配置继续读入 raw BGRA，不能消除公共 API 和运行时代码分支。
- **建议**：按设计边界拆出整改：先移除或私有化 raw BGRA 网络协议入口，再清理 host/client raw 运行分支和对应测试；如仍需保留本地解码后 BGRA 像素渲染，应改名为像素帧渲染边界，避免与网络 raw BGRA 协议混用。

### Host 会话线程 panic 仍会拖垮监听侧

- **分级**：高。
- **问题定性**：违反硬性工程基线。
- **依据**：`review-rust` 要求 `panic!` 限制在测试、初始化不可恢复路径或有明确说明的断言；长驻服务型入口不应让单会话异常升级为进程级崩溃。
- **证据**：`crates/wincast-host/src/agent/listener.rs:249` 至 `listener.rs:252` 和 `listener.rs:290` 至 `listener.rs:292` 都会对会话线程执行 `session.join()` 后调用 `log_session_result`；`listener.rs:303` 至 `listener.rs:317` 的 `log_session_result` 在 join 返回 `Err(_)` 时直接 `panic!("客户端会话线程异常结束且无法恢复会话资源")`。
- **风险**：任一客户端会话线程内部 panic 都会结束宿主端监听循环。后续进入 Windows Service/Agent 常驻形态后，这会把单连接问题扩大为宿主端不可用。
- **建议**：把会话线程 panic 收敛为会话级错误、状态上报和清理结果；监听循环继续运行。若当前作用域借用了 runner/locator/capture 导致 panic 后无法安全取回资源，应把资源所有权结构从 scoped borrow 调整为可恢复 owner，而不是在监听侧继续 panic。

### OpenH264 测试通过但反复输出码率控制 warning

- **分级**：中。
- **问题定性**：规范缺失导致风险。
- **依据**：`review-rust` 要求编译 warning、测试 warning 和 build script warning 作为风险记录；媒体后端配置属于可观测运行契约。
- **证据**：`cargo test --workspace --all-targets --all-features` 成功，但多次输出 OpenH264 warning：`bEnableFrameSkip = 0, bitrate can't be controlled ... without enabling skip frame`。对应编码器配置在 `crates/wincast-media/src/encoder.rs:19` 至 `encoder.rs:27` 设置 `.bitrate(...)`、`.rate_control_mode(openh264::encoder::RateControlMode::Bitrate)` 和 `.skip_frames(false)`。
- **风险**：测试虽通过，但后端已经提示当前参数组合下码率控制不可按预期工作。对于本项目的低延迟视频链路，这会影响带宽上限判断、烟测日志可读性和后续性能定位。
- **建议**：确认 OpenH264 当前版本的 bitrate + frame skip 约束，选择明确策略：要么允许 skip frame 并把低延迟影响写入测试和文档，要么改用不触发该 warning 的 rate control 配置，并补一条防止 warning/参数漂移的媒体边界测试或文档说明。

### 共享会话状态用锁中毒 expect 作为公共边界

- **分级**：中。
- **问题定性**：规范缺失导致风险。
- **依据**：`review-rust` 要求 `unwrap`、`expect`、`panic!` 限制在测试、初始化不可恢复路径或有清楚说明的断言；状态管理属于 host 运行期基础设施。
- **证据**：`crates/wincast-host/src/session_state.rs:190` 至 `session_state.rs:194` 的 `apply`、`session_state.rs:197` 至 `session_state.rs:201` 的 `remote_session_status`、`session_state.rs:204` 至 `session_state.rs:209` 的 `apply_detected_desktop_session` 都对 `RwLock` 使用 `.expect("shared session state lock should not be poisoned")`。
- **风险**：一旦会话状态写锁内发生 panic，后续状态查询和桌面事件应用会继续升级为进程 panic。该状态正服务于锁屏、登录和客户端准入判断，公共方法需要可报告错误或恢复策略。
- **建议**：将锁中毒转换为明确错误或保守拒绝状态，并把日志/状态上报放在 host 层处理；如果团队确认锁中毒必须视为进程不可恢复，需要在该模块文档中写清楚前提和理由。

### Windows FFI unsafe 前提仍分散

- **分级**：中。
- **问题定性**：规范缺失导致风险。
- **依据**：`review-rust` 要求 unsafe 集中、最小化，并写清楚安全前提。
- **证据**：`crates/wincast-host/src/window.rs:169` 至 `window.rs:198` 在窗口枚举 callback 中解引用 `lparam` 并调用多个 Win32 API；`crates/wincast-host/src/session_events.rs:147` 至 `session_events.rs:185` 读取 WTS buffer 并转为 `WTSINFOEXW`；`crates/wincast-input/src/input/injector.rs:34` 至 `injector.rs:81` 调用 `GetSystemMetrics` 和 `SendInput`。相比之下，`crates/wincast-capture/src/windows_impl.rs:200` 至 `windows_impl.rs:205` 已对裸指针转 slice 的 `SAFETY` 前提做了局部说明。
- **风险**：当前 FFI 前提散落在业务模块，指针来源、buffer 尺寸、句柄有效性和回调生命周期依赖读者推断。后续继续扩展 Service/Agent、窗口选择和输入注入时，unsafe 审计成本会上升。
- **建议**：优先把窗口枚举、WTS session 读取和输入注入各自收敛成小型平台适配函数，并补充 `// SAFETY:` 注释，说明指针来源、生命周期、线程/回调前提和失败语义。

### 渲染抽象命名仍混用 raw BGRA 协议语义

- **分级**：低。
- **问题定性**：建议优化。
- **依据**：`review-rust` 要求 crate、module、trait、struct、feature、配置键和公开 API 命名统一；当前设计已将网络传输口径收敛到 H.264。
- **证据**：`crates/wincast-render/src/lib.rs:104` 至 `lib.rs:108` 暴露 `RawBgraRenderer`，`lib.rs:121` 至 `lib.rs:123` 的错误文案仍写 `raw BGRA 渲染帧无效`，`crates/wincast-render/src/sdl.rs:20` 定义 `SdlRawBgraRenderer`。
- **风险**：如果该 crate 表达的是“解码后的 BGRA 像素帧渲染”，当前命名会与已删除的 raw BGRA 网络协议混在一起，导致调用方误读稳定边界。
- **建议**：在删除 raw BGRA 网络协议后，同步把渲染侧命名调整为 pixel frame / BGRA pixel renderer 一类的本地像素格式语义。

## 整改优先级建议

### 第一阶段：恢复硬性质量门禁

- 修复 `crates/wincast-client/src/stream.rs` 的 `clippy::derivable_impls`，重新执行 `cargo fmt --all -- --check` 和 `cargo clippy --workspace --all-targets --all-features -- -D warnings`。
- 将本次 clippy 失败作为提交前阻断条件处理，不要继续叠加业务改动。

### 第二阶段：收敛协议和运行边界

- 按 `docs/design.md` 清理 raw BGRA 网络协议公共 API、配置 codec、host/client 运行分支和测试主线。
- 重新命名渲染侧本地像素帧抽象，避免和网络协议命名混淆。

### 第三阶段：加固常驻 Host 和平台 FFI

- 重构会话线程 panic 后的资源恢复方式，避免监听循环被单会话异常拖垮。
- 将 session state 锁中毒处理变成可报告状态或明确不可恢复策略。
- 为窗口枚举、WTS session 读取和输入注入补齐最小平台适配封装与 `SAFETY` 前提。

### 第四阶段：明确 OpenH264 参数策略

- 处理 OpenH264 bitrate/skip-frame warning，并把最终策略同步到媒体边界测试和开发文档。
- 在 Linux x86_64 与 aarch64/ARM64 真机补跑 SDL2 + OpenH264 烟测，确认构建、窗口渲染和输入回传不是仅在 Windows 单机验证通过。
