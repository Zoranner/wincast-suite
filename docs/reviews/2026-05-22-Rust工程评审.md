# WinCast Suite Rust 工程评审报告

## 评审范围

本次只评审 Rust 工程体系、Cargo workspace、crate 边界、配置与协议契约、Windows 捕获和输入封装、线程与资源生命周期、unsafe/FFI、CI/发布配置、文档一致性和质量门禁；不评价真实 Windows host 到 Linux client 的功能验收效果。

本次评审以 2026-05-22 当前 `master` 工作树为准。评审过程中未修改业务代码，只新增本评审报告。

## 评审依据

- `review-engineering` 通用工程基线：结论必须有文件、配置、命令输出或仓库状态证据；入口层应轻量，边界应可复用、可验证；文档、配置和代码应保持一致。
- `review-rust` Rust 专项清单：检查 workspace、crate 边界、features、错误处理、线程、unsafe/FFI、依赖治理、Cargo 锁文件、fmt/clippy/test/build/doc 和文档同步。
- 项目现场规则：Rust 修改后需要执行 `cargo fmt --all` 和 `cargo clippy --all-targets --all-features -- -D warnings`；不隐藏问题；不做补丁式短期修正。
- 当前仓库事实：根 `Cargo.toml`、各 crate `Cargo.toml`、`Cargo.lock`、`.gitignore`、`.gitattributes`、`.github/workflows/release.yml`、`README.md`、`docs/design.md`、`docs/development.md`、`docs/smoke-test.md`、源码和测试目录。

## 总体结论

当前 workspace 的 crate 划分基本清楚：`wincast-protocol` 承担配置和协议，`wincast-capture` 承担 Windows 捕获，`wincast-media` 承担 H.264 编解码，`wincast-input` 承担 Windows 输入映射和注入，`wincast-render` 承担 SDL2 渲染，`wincast-host` 与 `wincast-client` 负责两端运行时组合。最近一次整屏重构已经把窗口定位模块删除，把 capture 公共目标收敛为 `CaptureTarget::Screen`，也把 raw BGRA 网络主线从公开协议面移出到测试模块。

本轮自动化质量门禁全部通过，说明当前代码具备基本可编译、可测试、可格式化和可生成文档的工程基线。剩余风险集中在运行期契约而不是编译层面：Host 在启动延迟期间不能及时响应客户端退出；会话期间没有监控已启动程序的实际存活；Windows 进程清理只覆盖直接 child，不覆盖进程树；DXGI 输出选择写死 adapter 0/output 0；协议仍暴露不再使用的 `VideoReady` 和 `WindowNotFound`。

## 主要问题

### 启动延迟期间无法及时响应客户端退出

- **分级**：高。
- **问题定性**：违反项目已确认规范。
- **证据**：Host 会话在程序启动后直接执行阻塞 sleep，见 `crates/wincast-host/src/agent/session.rs:216` 至 `session.rs:218`；输入读取线程要到 `write_h264_encoded_stream` 内才启动，见 `session.rs:228` 至 `session.rs:236` 和 `crates/wincast-host/src/agent/stream.rs:77` 至 `stream.rs:93`；程序 cleanup 发生在 `run_started_session` 返回之后，见 `session.rs:66` 至 `session.rs:77`。文档要求客户端断开或停止会话时清理本次启动的程序，见 `docs/design.md:39` 和 `docs/design.md:313`。
- **风险**：如果 `program.startup_delay_ms` 配置较长，Linux 客户端在延迟期间关闭窗口或断开连接，Host 不会立即读取 `StopSession` 或感知断开，而是先完整等待延迟，再尝试启动捕获，之后才进入清理路径。对真实 Unity 全屏程序，这会表现为客户端已退出但宿主端程序仍继续运行一段时间，和当前生命周期要求不一致。
- **建议**：把启动延迟实现为可取消等待。会话启动后应尽早建立输入/控制读取或连接关闭监听；延迟期间收到 `StopSession`、`Goodbye`、连接断开、锁屏或会话门禁拒绝时立即中止并清理已启动程序。补充“启动延迟期间客户端退出会清理程序”的回归测试。

### 会话不监控目标程序存活

- **分级**：高。
- **问题定性**：违反项目已确认规范。
- **证据**：`run_started_session` 接收 `StartedProgram` 后只执行 `let _ = started;`，见 `crates/wincast-host/src/agent/session.rs:208` 至 `session.rs:217`；后续推流循环只检查 capture、输入 reader 和 session gate，见 `crates/wincast-host/src/agent/stream.rs:155` 至 `stream.rs:196`。设计文档把“目标程序关闭后客户端收到明确错误或会话结束事件”列为验收场景，见 `docs/design.md:437`。
- **风险**：目标程序启动后如果自行退出、崩溃或被本地用户关闭，Host 仍会继续捕获整块桌面并向客户端推流。客户端看到的是宿主桌面或残留画面，而不是明确的目标程序已退出错误。这会让烟测误判“连接仍在线”，也会让后续资源清理依赖客户端主动退出。
- **建议**：为 `StartedProgram` 增加非阻塞状态检查能力，在主会话循环中定期检查目标进程是否仍存活；目标程序退出时发送明确错误或 `Goodbye`，并执行统一 cleanup。测试层增加“目标程序提前退出会结束会话并允许下一次连接”的用例。

### 程序清理只覆盖直接 child，不覆盖进程树

- **分级**：中。
- **问题定性**：规范缺失导致风险。
- **证据**：Windows cleanup 只对 `StartedProgram.child` 执行 `try_wait`、`kill` 和 `wait`，见 `crates/wincast-host/src/program.rs:141` 至 `program.rs:160`；`StartedProgram` 只记录直接 `std::process::Child` 和 `process_id`，见 `program.rs:26` 至 `program.rs:48`。
- **风险**：如果配置程序是 launcher、脚本、批处理、Unity bootstrapper 或会再拉起子进程的外壳，直接 child 退出后实际目标程序可能仍在运行；直接 child 被 kill 时，子进程也不一定被连带终止。这和“Linux 上退出，宿主机也退出”的部署口径存在差距。
- **建议**：Windows host 启动配置程序时使用 Job Object 或等价进程组策略，把本次会话启动的进程树纳入统一生命周期；cleanup 时关闭 Job 或终止进程树。文档中也要明确是否要求配置路径必须是最终目标 exe，或系统负责进程树清理。

### DXGI 输出选择写死为 adapter 0/output 0

- **分级**：中。
- **问题定性**：规范缺失导致风险。
- **证据**：`primary_output()` 固定调用 `EnumAdapters1(0)` 和 `EnumOutputs(0)`，见 `crates/wincast-capture/src/windows_impl.rs:373` 至 `windows_impl.rs:380`。当前文档只声明单显示器整屏捕获，但没有声明必须位于 DXGI adapter 0/output 0。
- **风险**：单显示器机器也可能因为独显/核显、远程桌面残留、虚拟显示驱动或采集卡导致可用输出不在 adapter 0/output 0。当前实现会枚举失败或捕获错误输出，错误信息也只能说“枚举 Windows 桌面复制输出失败”，无法定位到实际可用输出选择问题。
- **建议**：枚举所有 adapter/output，选择 attached desktop 的有效输出；如果多个输出存在，稳定版可以明确拒绝并列出检测到的输出数量。至少应把“没有找到可用桌面输出”和底层 DXGI 枚举错误区分开，便于现场排障。

### 协议公开面仍保留旧消息和错误码

- **分级**：中。
- **问题定性**：建议优化。
- **证据**：`ControlMessage` 仍包含 `VideoReady`，见 `crates/wincast-protocol/src/message.rs:11` 至 `message.rs:13`；`ErrorCode` 仍包含 `WindowNotFound`，见 `message.rs:58` 至 `message.rs:67`；协议 helper 仍公开 `send_video_ready`，见 `crates/wincast-protocol/src/handshake.rs:100` 至 `handshake.rs:112`；客户端错误格式化仍有“宿主端窗口定位失败”，见 `crates/wincast-client/src/errors.rs:3` 至 `errors.rs:12`。当前稳定设计已经不维护窗口捕获、窗口标题定位和单独 VideoReady 阶段，见 `docs/design.md:7`、`docs/design.md:130` 至 `docs/design.md:137`。
- **风险**：虽然当前主链路不再发送这些消息，但它们仍是公开协议类型的一部分。后续客户端、测试或第三方调用方可能继续按旧状态机处理 `VideoReady` 或 `WindowNotFound`，使协议契约和稳定版真实能力不一致。
- **建议**：既然当前不要求向后兼容，建议删除 `VideoReady`、`send_video_ready`、`WindowNotFound` 以及对应客户端错误映射和测试；如果为了协议保留位必须存在，应在枚举上明确标记为废弃，并在文档中说明 Host 不再发送。

## 已恢复或已确认良好的部分

- `wincast-client` 已有 lib target，`src/main.rs` 只调用 `wincast_client::run_default_client()` 并处理退出码，旧评审中的客户端 bin 边界问题已恢复。
- `wincast-protocol::raw_frame` 当前受 `#[cfg(test)]` 限制，未继续作为公开协议模块导出。
- `SharedSessionState` 对锁中毒已改为保守处理，不再通过 `expect` 让运行期状态读取直接 panic。
- Host listener 会话线程 panic 会被捕获并转换为错误，不再直接拖垮监听循环。
- D3D readback 已有 RAII unmap guard 和针对 `from_raw_parts` 的 SAFETY 注释，unsafe 审计状态比早期评审更清楚。
- CI release workflow 已包含 fmt、clippy、test 门禁，并通过 tag 构建 Windows host 与 Linux x86_64/aarch64 client 包。

## 整改优先级建议

### 第一阶段：会话生命周期闭环

- 改造 `program.startup_delay_ms` 为可取消等待。
- 会话循环加入目标进程存活检测。
- 增加启动延迟期间退出、目标程序提前退出、目标程序退出后下一次连接的回归测试。

### 第二阶段：程序树清理

- Windows 侧引入 Job Object 或等价进程树管理。
- 明确配置程序必须是最终目标进程，还是允许 launcher 场景。
- 文档同步部署约束和清理语义。

### 第三阶段：捕获输出选择和协议瘦身

- 枚举 DXGI adapter/output，选择有效桌面输出或明确拒绝多输出。
- 删除或废弃 `VideoReady`、`WindowNotFound` 等旧稳定版外协议面。

### 第四阶段：真机验证

- 等 CI 产物出来后执行 Windows host 到 Linux x86_64 client 真机烟测。
- 在 Linux aarch64/ARM64 真机验证 SDL2、OpenH264、窗口创建、渲染和输入回传。
