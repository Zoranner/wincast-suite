# Unity 内嵌远控后端实施计划

## 目标

把 Unity 内嵌远控后端推进到第一阶段工程骨架：Rust native core 可编译、Unity package 可集成、协议配置边界明确、Host 具备单 Unity 进程启动和监控配置模型。第一阶段不要求真实 Unity 运行验证，也不启动 Unity Editor。

## 当前约束

- Rust 修改后必须执行 `cargo fmt --all` 和 `cargo clippy --all-targets --all-features -- -D warnings`。
- 不使用 `CARGO_TARGET_DIR` 绕开 Windows 构建产物锁。
- Unity 相关工作不启动 Unity Editor、不运行 batchmode、不写 `Library/`、`Temp/`。
- Node.js 命令使用 bun；本计划不需要 Node.js。
- 当前工作区已有未提交改动，实施时不得回退无关文件。

## 任务分解

### Rust native core 骨架

写入范围：

- `Cargo.toml`
- `crates/wincast-unity-native/Cargo.toml`
- `crates/wincast-unity-native/src/lib.rs`
- `crates/wincast-unity-native/src/config.rs`
- `crates/wincast-unity-native/src/error.rs`
- `crates/wincast-unity-native/src/ffi.rs`
- `crates/wincast-unity-native/src/runtime.rs`
- `crates/wincast-unity-native/tests/ffi_boundary.rs`

要求：

- 新 crate 加入 workspace。
- crate type 为 `cdylib` 和 `rlib`，方便测试。
- 提供 C ABI：create、start、submit_frame、poll_input、get_status、get_last_error、shutdown。
- 第一阶段允许 runtime 只完成配置解析、状态流转、帧校验和输入队列边界，不要求真实网络监听和编码。
- 测试先红后绿，覆盖无效配置、创建/关闭、无效帧、错误消息缓冲区。

### Unity package 骨架

写入范围：

- `unity/com.zoranner.wincast/package.json`
- `unity/com.zoranner.wincast/Runtime/WinCastUnityAgent.cs`
- `unity/com.zoranner.wincast/Runtime/Capture/FinalFrameCapture.cs`
- `unity/com.zoranner.wincast/Runtime/Capture/FrameReadbackQueue.cs`
- `unity/com.zoranner.wincast/Runtime/Input/RemoteInputGateway.cs`

- `unity/com.zoranner.wincast/Runtime/Native/WinCastNativeTypes.cs`
- `unity/com.zoranner.wincast/README.md`

要求：

- 不创建 Unity 生成目录和 `.meta`。
- C# 代码只做 package 源码骨架，不依赖具体项目场景。
- `FinalFrameCapture` 使用 `WaitForEndOfFrame`、`ScreenCapture.CaptureScreenshotIntoRenderTexture` 和 `AsyncGPUReadback` 的结构。
- `WinCastNativeBridge` 只声明 P/Invoke 和安全包装，不包含业务逻辑。
- `RemoteInputGateway` 明确 Unity 主线程轮询输入。
- 不能启动 Unity Editor 验证；只做文本和结构检查。

- Host 第一阶段只解析和构造 Unity 固定端口启动参数；Unity 启动参数只允许 `--wincast-port <port>`。
- 配置测试覆盖默认 desktop、unity_embedded 解析、端口校验。

## 并行策略

- Rust native core 与 Unity package 可并行，写入范围完全分离。
- 协议和 Host 配置依赖 Rust native core 的配置命名，等 native core 返回后推进。
- 文档收口最后做，基于实际 diff 和验证结果。

## 第一阶段完成定义

- Workspace 包含 `wincast-unity-native`，可通过 Rust 格式和 clippy。
- Unity package 源码结构存在，能表达最终帧采集、native bridge 和输入网关职责。
- 协议配置能表达 `unity_embedded` 后端和单 Unity 进程参数。
- 文档明确这是第一阶段骨架，不声明 Unity 真机远控已跑通。

## 当前推进状态

- 已新增 `wincast-unity-native` crate，并加入 workspace。
- 已新增 `unity/com.zoranner.wincast` Unity package 骨架，未创建 `.meta`。
- 已补 `unity_embedded` 配置解析和 Host Unity 启动参数构造测试。
- Unity package 的 native bridge 已对齐 Rust 当前导出的 `wincast_unity_*` FFI 符号。
- Rust native core 已具备本地输入事件队列、轮询 ABI 和基础队列压缩策略；Unity package 已具备输入轮询和 UI 事件分发骨架。
- Rust native core 已补最小 TCP 会话监听骨架：可接受客户端 Hello/StartSession，发送 SessionReady，并把协议输入事件转换后放入 Unity 输入队列。
- `wincast_unity_submit_frame` 已在 native 调用内复制帧字节到 owned buffer，后续编码线程不依赖 Unity/C# 调用方 buffer 生命周期。
- Rust native core 已补最小 H.264 编码发送链路：会话建立后可读取 latest submitted frame，必要时 RGBA 转 BGRA，经 OpenH264 编码后发送 `EncodedVideoFrame`。
- 当前仍未完成 Unity Player 真机验证、端到端客户端验收、会话生命周期完善、单实例稳定性和 UI 输入验收。
- Host 启动 Unity 进程时只传入 `--wincast-port <port>`；Host 不再从配置读取或向 Unity 传入 token、额外标识、分辨率、FPS 或码率参数。
- Unity 进程的分辨率、FPS 和码率由 Unity package 或具体 Unity 项目自身配置，不由 Host 启动参数覆盖。
- 第一阶段不通过 Host、启动参数或 Rust native config 处理鉴权；后续如需要鉴权，再按单实例会话生命周期设计鉴权和授权。
- Rust native core 当前 last error 仍是进程全局状态，单实例骨架阶段可接受；后续如进入更复杂的并发会话模型，需要改成 handle 级或线程/调用级错误归属，避免错误状态互相覆盖。
- 当前未启动 Unity Editor、未运行 Unity 编译，也未执行 Unity 真机验证。
