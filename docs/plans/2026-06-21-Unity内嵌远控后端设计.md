# Unity 内嵌远控后端设计

## 背景

当前 WinCast 稳定版使用 Windows Host 启动目标程序，再通过 DXGI Desktop Duplication 捕获当前交互桌面整屏，并通过 Windows 输入注入把客户端键鼠操作回传给宿主机。这个路径适合普通 Windows 桌面程序，但不适合“远程控制 Unity 程序且不依赖物理显示器”的目标。

近期测试已经证明，真正关闭显示器会破坏 DXGI Desktop Duplication 的采集前提。目标程序又是可修改源码并重新打包的 Unity 程序，且界面以 Screen Space Overlay UI 为主，需要在单个 Unity Player 内完成远程画面输出和远程输入消费。因此新方案不应继续围绕显示器电源或桌面抓屏做补丁，而应在 Unity 进程内部采集最终画面，并在 Unity 内部消费远端输入。

## 目标

- 支持 Unity 程序内嵌远控输出，不依赖 Windows 桌面抓屏。
- 支持 Screen Space Overlay UI、多摄像机、后处理和复杂 UI 的最终画面采集。
- 支持单个 Unity Player 进程通过固定端口提供远控会话。
- 支持远端鼠标、键盘、滚轮、拖拽和文本输入，且不抢占本机系统鼠标键盘。
- 复用现有 WinCast 协议、H.264 编码、Linux 客户端渲染和输入事件模型。
- 让现有 `wincast-host` 从“采集和输入执行者”收缩为“单 Unity 进程启动和监控者”。

## 非目标

- 第一阶段不做虚拟显示器驱动。
- 第一阶段不做 Windows 桌面输入注入。
- 第一阶段只支持单 Unity Player 进程。
- 第一阶段不做多客户端同时控制同一个 Unity Player。
- 第一阶段不做公网访问、NAT 穿透、账号体系、审计日志或权限分级。
- 第一阶段不做 native GPU texture 直通硬件编码；先用 CPU H.264 编码验证闭环。
- 第一阶段不保证无侵入兼容所有旧 `Input.GetKey()` / `Input.GetMouseButton()` 业务代码。

## 总体架构

系统拆成三层：

```text
wincast-host
  -> 读取配置
  -> 拉起一个 Unity 进程
  -> 传入 --wincast-port <port>
  -> 前台监控 Unity 进程生命周期

Unity App + WinCast Unity Package
  -> 采集 Unity 最终 Game View 帧
  -> 把原始帧提交给 Rust native core
  -> 轮询远端输入事件
  -> 在 Unity 主线程分发 UI 和场景操作

Rust native core
  -> TCP 监听和协议握手
  -> 帧队列、限流和丢帧
  -> H.264 编码
  -> 输入事件接收和排队
  -> 会话状态和错误暴露
```

现有 Linux client 尽量保持不变，继续连接指定 host/port，接收 `SessionReady` 和 `EncodedVideoFrame`，并发送 `InputEvent`。

## Rust 工程划分

新增一个 native crate：

```text
crates/wincast-unity-native
```

该 crate 的 crate type 为 `cdylib`，Windows 产物为：

```text
wincast_unity_native.dll
```

它依赖并复用：

- `wincast-protocol`：握手、控制消息、输入事件和编码帧协议。
- `wincast-media`：H.264 编码、帧格式和媒体配置校验。

它不依赖：

- `wincast-capture`，因为 Unity 后端不使用 DXGI Desktop Duplication。
- `wincast-input`，因为 Unity 后端不使用 Windows `SendInput`。
- `wincast-host`，避免 native core 反向依赖应用层。

## Rust native core 职责

Rust native core 负责高性能、跨线程和网络相关工作：

- 创建和销毁单 Unity Player runtime。
- 在指定端口监听一个客户端连接。
- 执行 WinCast 协议握手。
- 发送 `SessionReady`。
- 接收 Unity 提交的原始帧。
- 对原始帧做尺寸校验、格式转换和 H.264 编码。
- 控制帧率、码率和关键帧请求。
- 使用有界队列保留最新帧，编码或网络慢时丢弃旧帧。
- 接收客户端 `InputEvent`，放入输入事件队列。
- 暴露 runtime 状态、错误和统计信息给 Unity。
- 在 Unity Player 关闭时清理网络线程、编码器和队列。

Rust native core 不负责：

- 调用任何 Unity API。
- 执行 UI raycast 或场景控制。
- 注入 Windows 系统输入。
- 拉起 Unity 进程。

## FFI 边界

Unity C# 通过 P/Invoke 调用 `wincast_unity_native.dll`。FFI 只暴露稳定 C ABI，不暴露 Rust 类型。

建议的第一版 API：

```c
typedef uint64_t WincastUnityHandle;

WincastResult wincast_unity_create(
    const char* config_json,
    WincastUnityHandle* out_handle
);

WincastResult wincast_unity_start(WincastUnityHandle handle);

WincastResult wincast_unity_submit_frame(
    WincastUnityHandle handle,
    const uint8_t* bytes,
    uint32_t width,
    uint32_t height,
    uint32_t row_pitch,
    WincastPixelFormat format,
    uint64_t timestamp_ns
);

uint32_t wincast_unity_poll_input(
    WincastUnityHandle handle,
    WincastInputEvent* out_events,
    uint32_t max_events
);

WincastRuntimeStatus wincast_unity_get_status(WincastUnityHandle handle);

uint32_t wincast_unity_get_last_error(
    WincastUnityHandle handle,
    char* out_utf8,
    uint32_t out_len
);

void wincast_unity_shutdown(WincastUnityHandle handle);
```

配置用 JSON 字符串传入，避免第一阶段为 C ABI 维护过多结构体字段。示例：

```json
{
  "listen": "127.0.0.1:7856",
  "video": {
    "width": 1280,
    "height": 720,
    "fps": 30,
    "bitrate_kbps": 4000,
    "max_bitrate_kbps": 8000
  }
}
```

## Unity Package 划分

新增 Unity 包：

```text
com.zoranner.wincast
```

建议目录结构：

```text
Packages/com.zoranner.wincast/
  package.json
  Runtime/
    WinCastUnityAgent.cs
    Capture/
      FinalFrameCapture.cs
      FrameReadbackQueue.cs
    Input/
      RemoteInputGateway.cs
      RemoteInputAdapter.cs
      UiEventDispatcher.cs
    Native/
      WinCastNativeBridge.cs
      WinCastNativeTypes.cs
  Plugins/
    x86_64/
      wincast_unity_native.dll
  Samples~/
    BasicRemoteScene/
```

Unity 包负责：

- 从 Host 启动参数读取 `--wincast-port <port>`，并从 Unity 组件或项目配置读取画面、帧率、码率等内部配置。
- 创建和管理 Rust native runtime。
- 在 Unity 渲染完成后采集最终帧。
- 把帧提交给 Rust native core。
- 在 Unity 主线程轮询远端输入。
- 把远端输入分发给 UI EventSystem 和业务输入适配器。
- 暴露 Inspector 配置、运行状态和错误信息。

Unity 包不负责：

- H.264 编码实现。
- TCP 协议细节。
- Unity 进程拉起。

## 画面采集设计

因为目标 Unity 程序以 Screen Space Overlay UI 为主，第一阶段不使用 URP Camera color target 作为主路径。URP Render Pass 只能稳定覆盖相机渲染结果，容易漏掉 Overlay Canvas。第一阶段采用最终 Game View 捕获路径：

```text
WaitForEndOfFrame
  -> ScreenCapture.CaptureScreenshotIntoRenderTexture
  -> AsyncGPUReadback.Request
  -> NativeArray<byte>
  -> wincast_unity_submit_frame
```

这个路径的目标是捕获 Unity 已经合成到 Game View 的最终结果，包括：

- 多摄像机最终显示结果。
- Screen Space Overlay UI。
- Screen Space Camera UI。
- World Space UI。
- 后处理后的画面。

采集策略：

- 固定输出分辨率由 Unity agent 配置决定，第一阶段建议 1280x720 / 30 FPS。
- 如果 readback 或编码队列仍繁忙，跳过当前帧，不堆积。
- 提交给 Rust 的像素格式第一阶段使用 BGRA8 或 RGBA8，具体以 Unity 读回格式和 `wincast-media` 转换成本最低者为准。
- 每帧带 `sequence_number` 和 `timestamp_ns`，用于编码、日志和客户端诊断。
- Unity 侧不得在主线程等待 Rust 编码完成。

后续性能优化方向：

- 使用 URP Final Blit 后的 color target 复制，前提是验证 Overlay UI 不丢失。
- 使用 native graphics plugin 获取 backbuffer 或 texture。
- 使用 Media Foundation / NVENC / AMF / QSV 做硬件编码。

这些优化不得改变第一阶段对最终画面完整性的验收口径。

## 输入设计

Unity 内嵌后端不使用 Windows `SendInput`。客户端输入进入 Rust native core 后，由 Unity 在主线程轮询并分发。

输入流：

```text
Linux client
  -> ControlMessage::InputEvent
  -> Rust native core input queue
  -> Unity Update() poll
  -> RemoteInputGateway
  -> UI EventSystem / RemoteInputAdapter
```

UI 输入分发：

- 鼠标位置按视频尺寸映射到 Unity 屏幕坐标。
- 鼠标按下、释放、点击、拖拽和滚轮通过 Unity EventSystem 分发。
- Button、Toggle、Slider、Dropdown、ScrollRect、InputField 和 TMP_InputField 是第一阶段重点。
- 文本输入需要补协议表达，不能只依赖 KeyCode。第一阶段可以新增文本输入事件，或在 Unity package 内把可打印按键转换为文本提交，但正式方案应走协议字段。

场景输入分发：

- 业务场景操作不直接伪造 Unity 旧 `Input` 静态状态。
- 新增 `RemoteInputAdapter`，把远端输入转成项目控制命令，例如相机旋转、平移、缩放、选择和快捷键。
- 项目中直接读 `Input.GetKey()`、`Input.GetMouseButton()` 的逻辑，应逐步迁移到本地输入和远端输入都可驱动的输入适配层。

本地输入和远端输入的关系：

- 第一阶段允许本地和远端输入同时存在，但远端输入只影响接入 `RemoteInputGateway` / `RemoteInputAdapter` 的 UI 和业务逻辑。
- 如需避免本地误操作，应由 Unity 项目自身提供本地输入锁定或观察者模式，不由 Rust native core 控制系统输入。

## 单实例设计

当前方案只面向一个 Unity Player 进程。Host 从配置读取固定端口，拉起 Unity 进程，并把该端口传给 Unity package：

```text
Unity Player -> port 7900
```

Host 拉起 Unity 时只允许传入端口：

```text
--wincast-port 7900
```

分辨率、FPS、码率和鉴权策略由 Unity package 或具体 Unity 项目内部配置，Host 不通过启动参数或环境变量覆盖。

单实例边界要求：

- Rust native core 在一个 Unity Player 内只维护一个 active runtime。
- FFI 返回的 handle 必须唯一映射到当前 runtime。
- runtime 拥有自己的监听 socket、编码器、帧队列、输入队列和状态。
- 日志需要能按 Unity 进程和端口定位。
- Unity Player 退出时，Host 以进程退出作为会话结束边界。

## Host 单进程监控设计

现有 `wincast-host` 在 Unity 内嵌模式下不再采集画面，也不注入输入。它负责读取配置、启动一个 Unity 进程，并在前台监控该进程：

- 读取 host 配置。
- 拉起 Unity exe，并只传入 `--wincast-port <port>`。
- 前台监控 Unity 进程状态。
- 在 Unity 退出时返回明确状态。

Host 配置第一阶段按以下模型扩展：

```toml
listen = "0.0.0.0:7856"
mode = "unity_embedded"

[program]
path = 'C:\Program Files\SomeUnityApp\UnityApp.exe'
args = []
work_dir = 'C:\Program Files\SomeUnityApp'
startup_delay_ms = 0

[unity]
executable = 'C:\Program Files\SomeUnityApp\UnityApp.exe'
work_dir = 'C:\Program Files\SomeUnityApp'
port = 7900

[video]
width = 1280
height = 720
fps = 30
codec = "h264"
bitrate_kbps = 4000
max_bitrate_kbps = 8000
```

稳定版可以继续保留 `desktop_dxgi` 模式服务普通程序：

```text
desktop_dxgi
  -> Host 负责启动程序、DXGI 抓屏、Windows 输入注入

unity_embedded
  -> Host 负责启动一个 Unity Player 并传入固定端口
  -> Unity Player 负责画面和输入
```

## 协议影响

第一阶段尽量复用现有控制消息：

- `Hello`
- `StartSession`
- `SessionReady`
- `EncodedVideoFrame`
- `InputEvent`
- `StopSession`
- `Heartbeat`
- `Goodbye`
- `Error`

Unity native core 可以扮演“Host 视频会话端”，直接和 Linux client 进行同一套协议交互。

需要新增或扩展的协议点：

- 文本输入事件，用于 TMP_InputField / InputField。
- 可选 `BackendInfo` 或 session metadata，用于客户端显示当前后端是 `unity_embedded`。
- 可选输入坐标基准，明确鼠标坐标相对视频帧、窗口客户区还是归一化空间。

协议扩展必须保持向后兼容。不能让现有 desktop_dxgi 客户端路径被 Unity 后端字段破坏。

## 队列与线程模型

Unity 主线程：

- 调用 `WaitForEndOfFrame` 后发起最终帧捕获。
- 提交已完成 readback 的帧指针给 native core。
- 每帧调用 `wincast_unity_poll_input` 拉取输入。
- 分发 UI 和场景操作。

Rust native core 线程：

- listener thread：等待客户端连接和握手。
- network reader：读取客户端输入和控制消息。
- encoder worker：取最新原始帧并编码。
- network writer：发送 `SessionReady`、`EncodedVideoFrame`、错误和心跳。

画面队列：

```text
Unity submit_frame
  -> latest raw frame slot, size 1
  -> encoder worker
  -> encoded frame send
```

规则：

- 新帧到来时，如果旧帧还没编码，可以替换旧帧。
- 不允许无界队列。
- 编码慢时优先降低实际 FPS，而不是增加延迟。
- 网络慢时可以丢帧，但必须保留关键帧请求能力。

输入队列：

```text
network reader
  -> bounded input queue
  -> Unity poll_input
```

规则：

- 输入队列必须有容量上限。
- 鼠标移动可以合并，只保留最新位置或累积 delta。
- 鼠标按下、释放、滚轮、键盘按下和释放不能随意丢弃。
- 队列溢出时必须记录状态，必要时向客户端发送错误或断开会话。

## 错误处理

Rust native core 需要区分：

- 配置无效。
- 端口绑定失败。
- 客户端协议版本不兼容。
- Unity 提交帧格式无效。
- 编码器初始化失败。
- 编码失败。
- 网络写入失败。
- 输入队列溢出。
- runtime 已关闭。

Unity package 需要区分：

- native dll 缺失或加载失败。
- native runtime 创建失败。
- final frame capture 不可用。
- AsyncGPUReadback 失败。
- submit frame 失败。
- poll input 失败。
- UI 分发失败。

错误信息面向用户时使用中文；底层日志保留进程、端口、序列号和 backend 名称。

## 安全边界

第一阶段仍按受控内网工具处理：

- 第一阶段不把 token 作为已实现或必需项，不通过 Host、启动参数或 native config 处理鉴权。
- Host 不保存 Windows 登录密码。
- Unity native core 不提供任意文件访问、命令执行或脚本注入能力。
- 远端输入只进入 Unity package 暴露的输入网关，不直接控制系统桌面。
- 如果实例端口绑定到 `0.0.0.0`，部署文档必须要求防火墙限制访问来源。

## 兼容与迁移

迁移分两条线：

```text
desktop_dxgi
  -> 保留给普通 Windows 程序
  -> 继续使用 wincast-capture 和 wincast-input

unity_embedded
  -> 面向可改源码的 Unity 程序
  -> 使用 Unity package 和 wincast-unity-native
```

客户端优先复用现有实现。必要时只增加：

- 文本输入事件。
- 后端信息展示。
- 鼠标坐标基准明确化。

不要为了 Unity 后端重写 Linux SDL2 渲染主链路。

## 实施阶段

### 原型闭环

- 新增 `wincast-unity-native` crate，输出 DLL。
- Unity package 能加载 DLL、创建 runtime、启动监听。
- Linux client 可以连接 Unity Player 端口，完成 Hello/StartSession/SessionReady。
- 使用测试帧或纯色帧验证编码和显示。

### 最终帧采集

- Unity package 接入 `ScreenCapture.CaptureScreenshotIntoRenderTexture`。
- 使用 `AsyncGPUReadback` 异步读回。
- 将最终 Game View 帧提交给 native core。
- 远端能看到包含 Overlay UI 的 Unity 画面。

### 输入闭环

- native core 接收客户端 `InputEvent`。
- Unity `RemoteInputGateway` 轮询输入。
- UI EventSystem 覆盖点击、拖拽、滚轮和基础文本输入。
- `RemoteInputAdapter` 覆盖场景相机和业务快捷键。

### Host 单进程监控

- `wincast-host` 增加 `unity_embedded` 模式。
- Host 读取 `[unity].port`。
- Host 拉起 Unity 并只传入 `--wincast-port <port>`。
- Host 前台监控 Unity 进程退出和资源释放。

### 单实例稳定性

- 单 Unity Player 长时间运行。
- 验证客户端断开、重连、StopSession 和 Unity 进程退出路径。
- 验证 CPU、GPU、内存、网络和延迟表现。

### 性能优化

- 评估 720p/30 FPS 和 1080p/30 FPS。
- 如 CPU 编码或 readback 成为瓶颈，再设计 native graphics plugin 或硬件编码后端。
- 优化必须在保持 Overlay UI 完整采集的前提下进行。

## 验收标准

- 单 Unity Player 1280x720 / 30 FPS 持续运行 30 分钟，画面无明显堆积，内存不持续增长。
- Overlay UI、Screen Space Camera UI、World Space UI 和多摄像机最终画面在远端完整显示。
- Button、Toggle、Slider、Dropdown、ScrollRect、拖拽、滚轮和文本输入可远端操作。
- 场景相机旋转、平移、缩放、选择和业务快捷键可远端操作。
- 本地鼠标键盘不被远端控制抢占。
- 客户端断开后 Unity Player 释放对应会话状态，并可接受下一次连接。
- Host 能感知 Unity Player 退出并返回明确状态。
- StopSession、客户端异常断开和 Unity 进程退出都能进入明确资源释放路径。

## 主要风险

- `ScreenCapture.CaptureScreenshotIntoRenderTexture` 能覆盖 Overlay UI，但性能可能低于 native backbuffer 方案；第一阶段必须以真机数据确认。
- `AsyncGPUReadback` 会引入帧延迟；需要通过丢帧和最新帧队列控制总体延迟。
- 旧 `Input` 静态 API 不适合作为远端输入注入目标；复杂业务代码需要接入 `RemoteInputAdapter`。
- CPU H.264 编码、readback 和 UI 输入分发可能成为单实例稳定性瓶颈；需要通过真机数据决定是否进入硬件编码或 native graphics plugin 优化。
- Unity package、native DLL 和 host 的版本需要一致，否则 FFI ABI 和协议字段可能不匹配。

## 待确认事项

- Unity 项目最低版本和 URP 版本。
- 是否使用新 Input System、旧 Input Manager，还是自研输入层。
- 需要支持的 UI 组件清单，尤其是 TMP_InputField、复杂拖拽和自定义控件。
- 单 Unity Player 目标机器 CPU/GPU 配置。
- 客户端是直接连接 Unity Player 端口，还是先连接 Host 再由 Host 管理启动状态。
- 后续是否需要 token 鉴权，以及鉴权和授权边界放在 Host、Unity package 还是客户端协议层。
